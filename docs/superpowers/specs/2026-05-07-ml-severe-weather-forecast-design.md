# ML Severe-Weather Forecaster — Design

**Status:** Approved (awaiting implementation plan)
**Author:** Charlie + Claude (brainstormed)
**Date:** 2026-05-07

## 1. Goal

Build a research-quality severe-weather ML forecaster following [Nadocast's](https://nadocast.com) methodology: probabilistic 24-hour severe-weather forecasts (tornado, severe hail, severe wind) over CONUS, generated by gradient-boosted decision trees trained on HRRR forecast fields and verified against SPC storm reports. The artifact is a public GitHub repository with a reproducible pipeline, a written verification report, and forecast-map figures suitable as a portfolio piece.

The work is historical-evaluation-only: no live HRRR ingestion, no operational deployment.

## 2. Non-goals

- Operational real-time forecasting.
- Beating Nadocast or SPC on absolute skill (a stretch goal at best).
- Multi-day outlooks (Day-2, Day-3+).
- Convection-allowing alternatives to HRRR (RRFS, FV3, etc.) as input.
- Neural-network or transformer models. (See Section 11 for why GBDT was chosen.)
- Web app / interactive dashboard.

## 3. Scope at a glance

| Dimension | Choice |
|---|---|
| Hazards | Tornado, severe hail (≥1"), severe wind (≥58 mph) — three independent models |
| Forecast horizon | Day-1: probability within next 24h, issued from each 12z HRRR run |
| Spatial resolution | 50 km Lambert Conformal grid over CONUS (~6,500 cells) |
| Temporal coverage | Apr 1 – Jul 31, 2022–2024 (3 severe seasons, 366 cycles total) |
| ML approach | XGBoost binary classifier per hazard, GPU-accelerated (`tree_method=hist`, `device=cuda`) |
| Calibration | Isotonic regression per hazard, fit on each fold's inner-validation set (Jul of training years) |
| Verification | 3-fold leave-one-year-out CV (no untouched holdout) |
| Comparison baselines | Climatology, raw `MXUPHL` thresholding, SPC Day-1 Convective Outlooks |
| Compute | Local: Ryzen 7 5800X / 32 GB / RTX 4070 SUPER / NVMe |
| Language / packaging | Python 3.12+, `uv` for dependency management |

## 4. Architecture

Single Python package `ml_severe_weather_forecast` with a typer CLI. Pipeline is six idempotent stages, each invokable independently:

```
download → extract → label → train → verify → plot
```

Each stage reads from / writes to canonical paths under `data/`. Stages communicate via Parquet on disk — no in-memory pipeline orchestration. Re-running a stage is the orchestration model.


`scripts/00_download_all.py` runs the three `download` subcommands in sequence; the rest of the orchestration scripts likewise wrap the per-stage commands.

## 5. Data layer

### 5.1 HRRR

- **Library:** `herbie-data`, which fronts AWS Open Data (`noaa-hrrr-bdp-pds` bucket) and handles GRIB2 lazy-reads via `cfgrib`.
- **Cycles:** 12z only, dates Apr 1 – Jul 31, 2022–2024 → 366 cycles.
- **Per-cycle inputs:** `wrfsfcf{FF}.grib2` for `FF ∈ {00, 01, …, 24}` (25 files/cycle).
- **Variables extracted** (~30 fields, partitioned into "extracted from HRRR" vs "computed at extraction time from extracted fields"):
  - **Extracted directly:** SBCAPE, MLCAPE, MUCAPE, SBCIN, MLCIN, lifted index, SRH 0–1km, SRH 0–3km, BWD 0–6km, BWD 0–1km, PWAT, T2m, Td2m, LCL height (`HGT_lvl_LCL`), pressure-level T at 850/700/500 mb, 500 mb height, 500 mb U/V, 500 mb absolute vorticity, 250 mb U/V
  - **Hourly-max storm-attribute fields** (extracted directly): max 2–5 km updraft helicity (`MXUPHL`, used both as a feature and as the source of the hail-proxy baseline in §9.2 — it's one field used two ways, not double-counted), max 10 m wind speed, max 1 km AGL reflectivity, max hail diameter (`HAIL`)
  - **Computed at extract time:** lapse rates 700–500 mb and 850–500 mb (from pressure-level T), dewpoint depression (T2m − Td2m), 500 mb wind speed (from U/V)
- **Persistence:** Each cycle's variables are extracted and rewritten as a single 50 km–grid Parquet under `data/features/{YYYY}/{MMDD}.parquet`. Raw GRIB2 is a transient cache; can be purged after extraction. Net storage: ~1.5 GB total.
- **Resilience:** `herbie` retries S3 fetches; `mlswf download` is idempotent (skips already-present files) with a `--verify` flag for GRIB2 message-count checks.

### 5.2 SPC storm reports

- **Training & verification source:** SPC severe-weather database annual CSVs (`spc.noaa.gov/wcm/#data`); 2022 + 2023 finals + 2024 prelims.
- **Climatology source (separate download):** Same SPC database, years 2010–2021 (all final). Used in §9.2 to compute the climatological-baseline probability per (50 km cell × month). Stored under `data/reports/climo/`. ~12 CSVs, <100 MB total.
- **Schema kept:** `event_time_utc, lat, lon, hazard ∈ {tor, hail, wind}, magnitude`.
- **Severity filters:** hail ≥ 1.00", wind ≥ 50 kt (58 mph) measured *or* damage-implied, tornado any EF rating.
- **Dedup:** Cluster reports by 5-min × 10-mile windows; keep the most-severe within each cluster.
- **2024 caveat:** Prelim reports are noisier than finals — verification skill numbers will be slightly pessimistic. Documented in the report.

### 5.3 SPC Day-1 Convective Outlook archive

- **Source:** Archived shapefiles from `spc.noaa.gov/products/outlook/archive/`, 12z issuance, 2022–2024 Apr–Jul (matches our HRRR cycle window).
- **Format:** Per-hazard probabilistic outlook polygons (categorical: 2/5/10/15/30/45/60% for tornado; lower 5/15/30/45/60% bands for hail and wind).
- **Persistence:** Rasterize onto our 50 km grid at ingest time → `data/spc_outlooks/{YYYY}/{MMDD}.parquet` keyed by `cell_id × hazard`. ~1 MB per cycle.

### 5.4 50 km grid

- **Projection:** Lambert Conformal Conic (NCEP CONUS standard parameters).
- **Cell side:** 50 km.
- **Coverage:** `lon ∈ [-125, -65], lat ∈ [24, 50]`.
- **Cell ID:** integer `(i, j)` plus stable string `c_{i:03d}_{j:03d}` for joins.
- **3 km → 50 km aggregation:** for each variable, retain `max` and `mean` of the underlying HRRR cells inside each 50 km cell.

## 6. Feature engineering

Per `(cycle_init_time, cell_id)` row, ~500 columns produced from the per-cycle Parquet:

### 6.1 Temporal aggregation across the 24h forecast window

| Field type | Aggregations |
|---|---|
| Instantaneous (CAPE, shear, SRH, lapse rates, moisture) | `max`, `mean`, `p90` over forecast hours f00–f24 |
| Hourly-max storm-attribute fields (`MXUPHL`, max wind, max refl) | `max` over the window |
| Snapshot anchors | values at f06, f12, f18 |

### 6.2 Spatial neighborhood features

Three concentric rings around each cell (radii 0–40, 40–80, 80–160 km). For each ring × each base variable: `max`, `mean`. ~120 columns.

### 6.3 Composite indices

Computed from the temporal-aggregated values, then themselves temporally-aggregated:

| Index | Formula |
|---|---|
| EHI 0–3km | `MLCAPE × SRH3 / 160000` |
| Supercell composite (SCP) | `(MUCAPE/1000) × (SRH3/50) × (BWD6/20)` |
| Significant tornado parameter (STP) | `(MLCAPE/1500) × (SRH1/150) × (BWD6/12) × ((2000-LCL)/1000) × ((MLCIN+200)/150)` |

## 7. Labels

For each `(cycle_init_time, cell_id, hazard)`:

```
label = 1 IF ∃ report r:
    r.hazard == hazard
    AND haversine(r.lat/lon, cell.center) ≤ 40 km     # ≈ 25 miles
    AND cycle.start_12z ≤ r.event_time < cycle.start_12z + 24h
ELSE 0
```

Implementation: sklearn `BallTree` with haversine metric over cell centers; per-cycle batch query of reports in the 24h window. Auxiliary columns `event_count` and `max_magnitude` are computed and stored for verification weighting / significant-event filtering, but not used as training targets.

### 7.1 Class imbalance

| Hazard | Approx positive rate per cell-day |
|---|---|
| Tornado | ~0.3% |
| Severe hail (≥1") | ~1.5% |
| Severe wind (≥58 mph) | ~2.5% |

Handling: XGBoost `scale_pos_weight = neg_count / pos_count` per hazard. No undersampling. Imbalance-aware metrics (Brier, FSS, reliability) for evaluation — never raw accuracy.

## 8. Modeling

### 8.1 One model per hazard

Three independent XGBoost binary classifiers — `tornado`, `hail`, `wind`. Independent because (a) the dominant features differ markedly across hazards, (b) per-hazard calibration is required anyway, (c) a multi-output stacked model would overfit to wind (densest signal) at tornado's expense.

### 8.2 Cross-validation

3-fold leave-one-year-out, with each fold's training data further split into a fit set and a held-out validation/calibration set (no leakage between booster training and the test fold):

| Fold | Booster fit set | Inner validation / calibration set | Test set |
|---|---|---|---|
| F1 | Apr–Jun of 2023 + 2024 | Jul of 2023 + 2024 | Apr–Jul of 2022 |
| F2 | Apr–Jun of 2022 + 2024 | Jul of 2022 + 2024 | Apr–Jul of 2023 |
| F3 | Apr–Jun of 2022 + 2023 | Jul of 2022 + 2023 | Apr–Jul of 2024 |

The inner validation set (July of the two training years) does double duty: it's the early-stopping watcher during booster fit, and it's the dataset isotonic calibration is fit on (§8.4). Test-year data is **never seen** during any fitting step within that fold — it's only evaluated at the very end.

Hyperparameter selection: best `(max_depth, learning_rate)` per `(hazard, fold)` is the combo with lowest inner-validation logloss. We do not retrain a final booster on the full training years (Apr–Jul) — the early-stopped booster from the fit set is the model used for test-fold prediction. This trades a small amount of training-data efficiency for a clean, no-leakage methodology.

Final reported metrics: see §9.1. We report both pooled-across-folds (concatenate test-fold predictions, compute a single metric) and per-fold mean ± std (variance estimate).

### 8.3 Hyperparameters

Most fixed from severe-weather ML literature; two tuned via grid search.

| Hyperparameter | Value | Source |
|---|---|---|
| `tree_method` | `hist` | XGBoost recommended for tabular |
| `device` | `cuda` | RTX 4070 SUPER |
| `scale_pos_weight` | `neg/pos` per hazard per fold | imbalance handling |
| `eval_metric` | `logloss` | smooth, calibration-friendly |
| `early_stopping_rounds` | 50 | standard |
| `n_estimators` | 5000 (early-stopped) | upper bound |
| `subsample` | 0.8 | fixed, regularization |
| `colsample_bytree` | 0.6 | fixed, neighborhood-feature redundancy |
| `min_child_weight` | 5 | fixed, rare-event safeguard |
| `reg_lambda` | 1.0 | fixed, default |
| `max_depth` | **tune** ∈ {4, 6, 8, 10} | meaningful interaction-depth fork |
| `learning_rate` | **tune** ∈ {0.03, 0.05, 0.1} | trades off n_estimators |

12 HP combos × 3 hazards × 3 folds = 108 trainings. Estimated ~2–5 min each on the 4070 → 3–6 hours wall-clock.

### 8.4 Calibration

Isotonic regression (sklearn `IsotonicRegression(out_of_bounds="clip")`), fit per hazard per fold on the **inner validation / calibration set** (the July-of-training-years split, see §8.2). Procedure per fold:

1. Booster fits on `train_fit` (Apr–Jun of training years), early-stopped on `train_calib` (Jul of training years).
2. Predict raw probabilities on `train_calib`.
3. Fit isotonic regression: `IsotonicRegression().fit(raw_probs_calib, labels_calib)`.
4. Predict raw probabilities on `test_fold`, then transform via the fitted isotonic → calibrated probabilities.
5. All §9 metrics computed on these calibrated test-fold probabilities.

This avoids the leakage that would result from co-fitting calibration on the test fold. The mild double-use of `train_calib` for both early stopping and calibration is acknowledged but standard; its effect is dominated by the booster's regularization.

### 8.5 Outputs of training

Per `(hazard, fold)`: `data/models/{hazard}/fold{i}.joblib` containing `{booster, isotonic, train_curve, val_curve, reliability_data}`. Plus an aggregated `data/models/{hazard}.joblib` bundling all three folds for downstream verification.

## 9. Verification

### 9.1 Metrics

Each test-fold's calibrated probabilities (§8.4) are concatenated into a single pooled prediction set across all three folds, since each cycle has exactly one prediction (made when its year was the held-out test fold).

| Metric | Computed how |
|---|---|
| **Brier score** + decomposition (reliability, resolution, uncertainty) | Standard formula, 10 bins for decomposition |
| **Brier skill score** vs. climatology | `1 - BS_model / BS_climo` |
| **Reliability diagram** | 10 quantile bins |
| **ROC + AUC** | sklearn |
| **Performance diagram** (POD vs SR / CSI / FAR) | thresholds at every percentile of forecast probability |
| **Fractions skill score** | radii {40, 80, 160} km |

We report each metric two ways:

1. **Pooled** (primary headline number): computed on the concatenated 3-fold prediction set. Largest sample → tightest estimate.
2. **Per-fold mean ± std** (variance estimate): metric computed independently on each test fold, then averaged across folds with std reported. Tells us how stable the skill is year-to-year.

All metrics are also stratified by month (Apr/May/Jun/Jul) for the monthly skill-score heatmap (§9.3).

### 9.2 Baselines

1. **Climatology:** per (50 km cell × month) base rate computed from 2010–2021 SPC reports.
2. **Raw `MXUPHL` thresholding:** per-hazard sigmoid-fit `P(severe) = σ(a × MXUPHL_max + b)` with a/b tuned on 2022. Tests whether ML adds value over the most-relevant single feature.
3. **SPC Day-1 Convective Outlook (probabilistic, per-hazard):** archived shapefiles from `spc.noaa.gov/products/outlook/archive/`. SPC's per-hazard probabilistic outlook bins are tornado: {2, 5, 10, 15, 30, 45, 60}%; hail and wind: {5, 15, 30, 45, 60}%. The polygon-rasterized per-cell probability stored in `data/spc_outlooks/` (§5.3) is used directly with no mapping. Significant-severe hatched areas are ignored for this comparison.

### 9.3 Visualizations

**Per-cycle forecast maps** — PNG per `(hazard × cycle)` for verification cycles, all 3 fold-years (~3,300 images, ~700 MB total, gitignored, regenerable):
- Cartopy CONUS basemap (states + coast)
- Filled-contour probability at 50 km, color stops matching SPC's outlook palette (2/5/10/15/30/45/60%)
- Black dots for actual storm reports of that hazard within the 24h window
- Title: cycle init, hazard, fold-year, Brier score for that cycle
- Side-by-side panel: our forecast vs. SPC Day-1 outlook for the same day

**Static report figures** — committed to repo, regenerated by `mlswf verify`:
- Reliability diagram per hazard (3 panels)
- ROC + performance diagram per hazard
- Top-20 feature-importance bar chart per hazard (XGBoost gain)
- SHAP summary plot per hazard, computed on a 50k-row stratified sample
- Monthly skill-score heatmap (rows: hazards, cols: months, color: BSS)
- Hits/misses mosaic per hazard: best-skill day, worst-skill significant-event day, false alarm, surprise event

**Verification report** — `notebooks/04_verification_report.ipynb`, rendered to a static HTML page in `docs/`. Title, methodology summary, headline numbers vs. SPC, all figures with captions. The portfolio artifact.

## 10. Testing & error handling

### 10.1 Tests

Three tiers, all fast enough for CI:

- **Unit tests** with `tests/fixtures/tiny_hrrr.grib2` (one cycle, ~5 MB Oklahoma bbox) and `tests/fixtures/tiny_reports.csv` (one tornado day):
  - `test_grid.py` — projection roundtrips (<10 m), area conservation, cell-id idempotency
  - `test_features.py` — known-input → known-output, NaN handling
  - `test_labels.py` — known tornado labeled `1`; distant cells `0`; out-of-window reports excluded; dedup
  - `test_neighborhood.py` — ring-sum sanity
  - `test_calibration.py` — known miscalibrated input → expected monotone output
- **Integration test** `test_end_to_end.py` — full pipeline on the fixture, ~30s.
- **No tests for `viz.py`** — manual visual inspection.

### 10.2 Error handling

| Failure mode | Strategy |
|---|---|
| S3 timeout / partial download | herbie 3× retry w/ backoff; final failure → log + skip cycle, accumulate skip list, fail job if >5% skipped |
| Corrupt GRIB2 | Skip the entire cycle; do not half-fill features |
| SPC report parse error | Hard fail — labels are foundational |
| Missing field in GRIB2 | Hard fail with diagnostic — no silent fallback |
| XGBoost OOM | Pre-flight memory check, fail fast with clear message |
| Calibration on single-class data | Hard fail — upstream feature pipeline is broken |

Logging: `structlog` JSONL to `data/logs/{stage}-{YYYYMMDD}.jsonl`. Each CLI command emits a structured run-summary at end.

### 10.3 Repo hygiene

- **Pre-commit:** `ruff` (lint+format), `mypy` (strict on `src/`), `nbstripout`.
- **CI** (GitHub Actions): unit + integration tests on push.
- **Reproducibility:** every CLI command writes `meta.json` next to outputs (git SHA, package versions, args, runtime). Verification figures embed this.
- **`.gitignore`:** all `data/` except `data/.gitkeep`; notebook checkpoints; trained model artifacts (publish via GitHub Releases).

## 11. Decision log (why these choices)

- **GBDT over neural networks.** Faithful to Nadocast. CPU-trainable but GPU-accelerated on `hist`. Interpretable feature importance is a portfolio asset. Severe weather is a tabular-ish problem at 50 km grid; the spatial-pattern advantage of CNNs is largely captured by neighborhood-ring features.
- **3 years Apr–Jul over 5 years full-year.** Roughly equal severe-event count (peak season is concentrated), much less storage and processing time.
- **Independent models per hazard over multi-output.** See §8.1.
- **K-fold leave-one-year-out over untouched holdout.** User preference; tighter estimate at the cost of clean holdout. Documented in the report.
- **No spatial CV.** Goal is national skill comparable to SPC outlooks. Spatial CV would systematically penalize learned regional climatology that is itself useful information.
- **No workflow framework.** Pipeline is six idempotent CLI commands; orchestration is re-running them. Snakemake/Airflow at this scale is overhead with no payoff.
- **Notebooks for analysis only, not pipeline.** All logic in the package; notebooks call into it. Testable, reviewable, not subject to notebook-state hell.
- **`uv` over conda.** Faster, lockfile-based, modern. Conda fallback documented in README if `cfgrib`/`eccodes` cause Windows pain.

## 12. Out of scope (explicit non-goals revisited)

- Live HRRR ingestion / real-time forecasts.
- Operational deployment, monitoring, alerting.
- Web app / interactive map.
- Day-2+ outlooks.
- Model variants (RRFS, FV3, GFS) as input.
- Neural networks, transformers, foundation-model fine-tuning.
- Federated / distributed training.
- Docker / containerization.
- Experiment tracking (MLflow / W&B) — `meta.json` is sufficient.
